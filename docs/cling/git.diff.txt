... for 9000+ more lines of mostly additions to the codebase. It was unusable for my purposes.

So I wrote a deterministic tool called `changeling` (`cling` for short) that figures out what changes were important, which were wrongly named, which may have multiple concerns that should have been separated, etc. and presents it in a release-notes/changelog sort of way.

As I said, the tools is not "feed your git log to an LLM and what it make something pretty, but possiby inaccurate. No, I had a second motivation which is going to rub a lot of "repo source-of-truthers" the wrong way, and in some cases I would align with them, but in the general sense, it's just a tool - don't do things that erode trust and make it harder to collaborate with others; with great power and all that. But, that said, if I want to rewrite history to be more legible on a repo or replica that is entirely moine, I may not care about "truth" as much as I care about accuracy and legibility. I don't need documentation of mine or other's mistakes or lazy commit messages to know the truth; that sort of truth is just not always important. For an audited account of something where that is important, sure... but I'm not going to buy into the sanctity of a repo's commit ledger for the purposes of usability alone. If I can fork the code and do what I want with it, that ust include the ability to write fiction in git history, add dumb features, break the product, or take it out of revision control entirely and just copy it to floppies. Public source control is notT sacre.

So perhaps you've guessed where I am goingg with this. CLING also as a `speakwrite`/`speakright` mode, where history is changed so that the Parrty is always correct, and the "unpersons" are memory-holed.

But, the primary purpose, to generate less-embarrasing change set lists automatically is what i will demonstrate here.

But, this deterministically-generated data can be, optionally, send to an LLM in structured data and very strict prompting, to "enrich" the text based on a grouping of the commits based on similar deterministic criteria. So the release-notes you just saw, can be further refined with a pass from a small language model to summarize a cluster of similar changes into coherent, descriptive text.
